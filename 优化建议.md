我把 SAPI 理解为 **Minecraft Bedrock Script API**。看完后，我的总体判断是：这个项目功能已经很完整，真正最该补的不是继续堆玩法，而是把“服务器级管理插件”的底座稳定下来。

当前值得肯定的地方：已经做了标准版/BDS 增强版双构建；用了 SAPI 自定义命令`runJobinputPermissionsserver-admin` 进服前拦截`server-net` 动态查询`CustomForm` fallback、公会 facade 缓存、领地快照分片等。这说明项目方向是对的，已经在吃新版 SAPI 的红利。

**推荐优先级**

1. **P0：修复工程健康与发布一致性**

   

   现在 `npx tsc --noEmit` 能过，但 `npm run lint` 直接失败`eslint.config.mjs` 引用了 `@typescript-eslint/eslint-plugin`，而 `package.json` 没有这个依赖。README 也明显落后：还写“支持 1.21.70”“经济系统后期计划”，但项目实际已经有经济、公会、日志、反作弊、快照等。  

   优先理由：这会影响贡献者、CI、发布可信度。先补齐 lint 依赖或简化 ESLint 配置，同时更新 README、功能矩阵、普通版/BDS版差异说明。

2. **P0：抽一个 SAPI 能力边界层**

   

   现在 BDS/标准版分叉靠 `main.standard.tsmain.bds.ts__BDS_BUILD__`、动态 import 分散处理。建议加一层类似 `features/platform/sapi-capabilities`：集中包装 `server-adminserver-netCustomForm/Observable`、预览事件`runCommand` 能力检测。  

   优先理由：官方 `@minecraft/server-admin` 明确是 BDS-only 且仍是 pre-release，后续变更风险高。集中隔离后，黑名单、实时面板、XUID 查询、未来 allowlist/备份等功能都不会把兼容性判断散落到 UI 和业务服务里。

3. **P1：重构持久化层，做版本、迁移、备份、健康检查**

   

   [database.ts](C:\Users\27662\Documents\projects\mcbes-manage-script\scripts\shared\database\database.ts) 现在用动态属性分块保存整库 JSON，每秒自动保存 dirty 数据。这个思路适合标准版，但现在日志、经济、公会、领地、快照都在长大，风险会集中到这里。  

   建议做 `VersionedStore`：统一 `schemaVersion`、迁移钩子、旧 chunk 清理、容量预警、手动 flush、损坏恢复报告。BDS 增强版可利用 `@minecraft/server-admin` 的 `LevelStorage.saveHold/saveResume` 做“更新/恢复快照前世界存档快照”入口。  

   优先理由：这是所有功能的数据命门，收益比继续加新玩法更高。

4. **P1：建立统一调度器与性能面板**

   

   项目里 `system.runInterval` 很多：数据库自动保存、服务器实体统计、行为日志、领地检测、PVP、名称显示、红包过期、快照队列等。SAPI 官方也建议长任务用 `system.runJob` 分片，避免 watchdog 和单 tick 压力。  

   建议把高频任务登记到统一 scheduler：支持任务名、间隔、耗时统计、开关、降频策略，并把结果展示到实时面板。  

   优先理由：你现在已经有服主管理面板了，下一步最自然的是让它能看出“谁在耗 tick”。

5. **P1：拆大文件，尤其是命令和系统 UI**

   

   目前最大压力点很明显：[system/index.ts](C:\Users\27662\Documents\projects\mcbes-manage-script\scripts\ui\forms\system\index.ts) 2200+ 行，[command.ts](C:\Users\27662\Documents\projects\mcbes-manage-script\scripts\features\command\services\command.ts) 1600+ 行，[guild/index.ts](C:\Users\27662\Documents\projects\mcbes-manage-script\scripts\ui\forms\guild\index.ts) 1500+ 行。  

   建议按功能域拆：命令定义表、命令处理器、参数解析、权限判断；系统 UI 拆成经济/领地/日志/防刷/公告/图标修复子表单。SAPI 的 `CustomCommandRegistry.registerEnum` 也可以用于有限动作参数，减少自由字符串解析。  

   优先理由：现在不是不能维护，而是每次改动会越来越容易牵连无关功能。

6. **P1：建立统一身份服务**

   

   黑名单已经在用 `name + persistentId + xuid` 三层匹配，这是对的；但经济、领地、公会、路点大量仍以玩家名为主键。建议引入 `identityService`：维护 `name -> persistentId/xuid/lastSeenName`，BDS 用 `asyncPlayerJoin` 增强，标准版降级为名字。  

   优先理由：玩家改名会逐渐成为经济、公会、领地权限的一致性风险。黑名单已经解决了一半，最好把这套能力抽出来给全系统用。

7. **P2：新功能方向优先做“管理增强”，不是玩法堆叠**

   

   推荐的新功能顺序：

   - **审计中心**：把行为日志、物品监控、领地破坏尝试、黑名单、公会金库流水汇总成风险视图。

   - **BDS 运维面板**：allowlist 管理、kick/op/deop、reload scripting config、停服前提醒等，可基于 `server-admin`。

   - **Webhook 告警**：BDS 增强版用 `server-net` 把高危行为推送到 QQ/Discord/企业微信等。

   - **权限/RBAC**：从单纯 `tag admin` 升级为“服主/管理员/巡查/经济管理员/日志查看员”。

**我的推荐路线**

先做`工程健康 + README/发布说明 + SAPI 能力层`。  

然后做`持久化 VersionedStore + 统一调度器`。  

最后再做`审计中心 / BDS 运维面板 / Webhook 告警`。

这条路线比较适合现在的项目阶段：你已经有足够多功能了，下一步最值钱的是让它更稳、更容易发版、更能利用 BDS 专属 SAPI，而不是继续把新功能塞进几个巨型文件里。

验证结果`npx tsc --noEmit` 通过`npm run lint` 失败，原因是缺少 `@typescript-eslint/eslint-plugin` 依赖。
